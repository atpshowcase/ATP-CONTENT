Tiptap Style Notes - A to F .NET Learning Path

Fundamental .NET Up to Real Application Practice

Explained casually but structured, starting from .NET foundations, then project structure, Web API, up to real-world best practices.

A. Fundamental .NET

A.1 Introduction to the .NET Platform

.NET is Microsoft’s application development platform that you can use to build web, desktop, mobile, cloud services, and even background workers. So it’s not just the C# language, but a complete ecosystem: runtime, libraries, tooling, and CLI.

  • Runtime: the engine that runs your app.
  • Base Class Library (BCL): built-in functions so you don’t have to code from scratch.
  • SDK and CLI: tools to build, run, test, and publish.
  • Main languages: C#, F#, and Visual Basic.
// check installed .NET SDK
// dotnet --info

// create a new console project
// dotnet new console -n LearnDotNet

// run the project
// dotnet run
Analogy: .NET is like a complete sports complex. You can play futsal, basketball, or hit the gym in the same area because all the facilities are integrated.

A.2 Basic .NET Architecture

You can see .NET’s basic architecture as a layered flow: your C# code is compiled to IL, then executed by the CLR. The CLR handles memory management, garbage collection, type safety, and runtime security.

  1. Source Code (C#): the code you write.
  2. Compiler (Roslyn): turns code into Intermediate Language (IL).
  3. CLR (Common Language Runtime): runs IL, optimizes via JIT, manages memory.
  4. BCL/NuGet Libraries: extra functions for networking, files, JSON, database, and more.
// Simple flow:
// C# code -> IL -> CLR/JIT -> Machine Code

string name = "The .NET Platform";
Console.WriteLine($"Learning {name} from its basic architecture");
Analogy: it’s like a restaurant kitchen. You (the developer) write the order, the chef (compiler + CLR) processes it into ready-to-serve food for the machine.

A.3 Differences: .NET Framework, .NET Core, and .NET Modern

To make it simple, think of .NET’s evolution as three main phases. Now, the recommended choice for new projects is .NET Modern (version 6/7/8 and up).

  • .NET Framework: old generation, Windows-dominant, good for legacy enterprise apps.
  • .NET Core: transition generation, cross-platform (Windows, Linux, macOS), lighter performance.
  • .NET Modern (the product name is back to .NET): the official successor to .NET Core with LTS/STS, more mature tooling, and the most active ecosystem.

Analogy: .NET Framework is like an old motorcycle that still runs, .NET Core is the early upgrade version, and .NET Modern is the latest model—most efficient, fastest, and with the most repair shop support.

A.4 The History of .NET

The history of .NET started in the early 2000s when Microsoft introduced .NET Framework (2002) to unify how to build Windows and web apps. After that, the ecosystem grew fast until .NET Core (2016) was born—cross-platform and the foundation of modern .NET.

  • 2002: .NET Framework 1.0 officially released.
  • 2016: .NET Core 1.0 released, starting the cross-platform era.
  • 2020: .NET 5 unified the branding into one track.
  • 2021-now: .NET 6/7/8 focus on performance, cloud, and productivity.

B. Web Frameworks in .NET

B.1 Introduction to ASP.NET

ASP.NET is the web framework in the .NET ecosystem for building dynamic web apps, APIs, and real-time apps. Basically, if you want to build a web backend in .NET, ASP.NET is the most common entry point.

  • HTTP routing support (GET, POST, PUT, DELETE).
  • Built-in middleware pipeline.
  • Strong integration with authentication, logging, and DI.
  • Works for both monoliths and microservices.
// Minimal API (ASP.NET Core)
var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

app.MapGet("/", () => "Hello from ASP.NET!");

app.Run();
Analogy: ASP.NET is like the main kitchen of an online restaurant—every web order comes through here and gets processed neatly.

B.2 ASP.NET MVC

ASP.NET MVC uses the Model-View-Controller pattern so code is separated by responsibility: data, view, and request flow. This pattern is great for server-rendered web apps and teams that need clear structure.

  1. Model: represents data and business rules.
  2. View: the HTML the user sees.
  3. Controller: bridges user requests to app logic.
// Simple controller in ASP.NET MVC
public class HomeController : Controller
{
    public IActionResult Index()
    {
        ViewData["Message"] = "Welcome to ASP.NET MVC";
        return View();
    }
}
Analogy: MVC is like a content production team: the writer (Model), the layout editor (View), and the producer who manages the workflow (Controller).

B.3 ASP.NET Core

ASP.NET Core is the modern version—lighter, faster, and cross-platform. This is the main web framework used for new projects now because of its high performance and flexible deployment to Linux, Windows, containers, and the cloud.

  • Cross-platform: runs on Windows, Linux, macOS.
  • High performance and more efficient memory footprint.
  • Built-in dependency injection.
  • Supports MVC, Razor Pages, Web API, SignalR, Minimal API.
// Register services + endpoints in ASP.NET Core
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddControllers();

var app = builder.Build();
app.MapControllers();
app.Run();
Analogy: ASP.NET Core is like the latest operational vehicle: more efficient, faster, and can be used in more terrains.

C. Project Structure and Components

.NET Project Structure

.NET projects usually already have a folder pattern, so your team can easily follow the code flow even if the people change.

  • Controllers: entry point for user requests.
  • Models: data or entity representations.
  • Views: what the user sees (for MVC projects).
  • Config files: set up environment, dependencies, and app behavior.

Layer Architecture Concept

Layer architecture is a technique to keep code responsibilities from getting mixed up.

  1. Controller Layer: receives requests and directs the process.
  2. Service Layer: contains the main business logic.
  3. Repository Layer: handles data/database access.
  4. Domain Model: core data shape according to business needs.

Dependency and Package Management

In .NET, package management is usually done via NuGet. It’s like a package manager in other ecosystems, but specific to the .NET world.

  • NuGet is used to install third-party libraries.
  • Package versions need to be managed to avoid dependency conflicts.
  • You can add packages via CLI or Visual Studio.

D. Web API Development Basics

Web API Concept

An API is a communication bridge between applications. The frontend requests data, the backend gives a response.

  • REST APIs in .NET usually use endpoints based on HTTP methods (GET, POST, PUT, DELETE).
  • The most common data format: JSON.
  • APIs must be consistent so the frontend/mobile team can use them easily.

Creating a Simple API

Start simple so the flow is easy to follow.

  1. Create an API controller.
  2. Set up basic endpoint routing.
  3. Return a JSON response from the action method.
Hangout tip: don’t overthink complexity. A simple, tidy API is more useful than a fancy API that confuses the team.

E. Application Development Practice

Building a Simple Web App

At the practice stage, the focus isn’t on fancy features, but on building the right workflow.

  • Create a new project using the right template.
  • Add a controller to handle requests.
  • Show data in the view so the end-to-end flow is clear.

Connecting to a Database

If your app only stores data in variables, it’ll disappear when restarted. To make data truly persistent, you have to connect to a database. This is a super crucial topic.

ORM Concept

ORM stands for Object-Relational Mapping. The point: you write code using objects (like a Product or User class), and the ORM handles how that data is stored and read from database tables. You don’t have to write raw SQL for every operation.

Imagine you have a class like this:

public class Product
{
    public int Id { get; set; }
    public string Name { get; set; }
}

With ORM, this class is automatically mapped to a Products table in the database. The Id field becomes the Id column, Name becomes the Name column. You don’t have to write any SQL at all.

The benefit? You write code in C#, not SQL. More readable, more type-safe, and easier to maintain.

Introduction to Entity Framework

Entity Framework (EF) is Microsoft’s built-in ORM for .NET. It’s the most commonly used in the .NET world, and you’ll find it in almost every project.

There are two main approaches in EF:

  • Code First — you create the C# class first, then EF generates the database tables. Great for new projects.
  • Database First — you already have a database, then EF generates the C# classes from the existing tables. Great for legacy projects.

The most common now is Code First. You make a model, create a DbContext (this is like the “main entrance” to your database), then run migration to generate the tables.

The flow is simple: Create Model → Create DbContext → Add DbSet → Run Migration → Tables created. Done. Your database is ready to use.

Basic CRUD

CRUD is the four most basic operations you do to a database: Create, Read, Update, Delete. Almost every app feature revolves around these four things.

// CREATE — add new data
var product = new Product { Name = "Laptop" };
db.Products.Add(product);
db.SaveChanges();

// READ — get all data, or find one by ID
var all = db.Products.ToList();
var one = db.Products.Find(1);

// UPDATE — get first, modify, then save
var p = db.Products.Find(1);
p.Name = "Laptop Pro";
db.SaveChanges();

// DELETE — get first, remove, then save
var p = db.Products.Find(1);
db.Products.Remove(p);
db.SaveChanges();

You’ll notice the same pattern over and over: get from context → modify → save. That’s the core of how EF works.

Tip: Don’t skip manual CRUD practice first. Many people jump straight to auto-generated scaffolding, but if you don’t understand how EF works behind the scenes, you’ll get confused when there’s a bug or you want to optimize a query.

F. Closing

.NET Development Best Practices

As your project grows, best practices aren’t optional anymore—they’re a must.

  • Keep your project structure tidy so the team can maintain it easily.
  • Write clean code so logic is easy to read and test.
  • Use modular architecture to make scaling easier.
In short: strengthen your foundation first, then speed up to features. In .NET, having a neat architecture from the start gives a huge long-term impact.
