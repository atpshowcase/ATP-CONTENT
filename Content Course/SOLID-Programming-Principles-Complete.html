Materi lengkap SOLID dengan gaya santai, analogi tongkrongan, dan contoh kode C# yang langsung kepake di project nyata.

BAB A. Introduction to SOLID

A.1 What is SOLID?

SOLID itu kumpulan 5 prinsip desain OOP biar code lo rapi, gampang diubah, dan gak bikin pusing pas project makin besar.

// S = Single Responsibility
// O = Open Closed
// L = Liskov Substitution
// I = Interface Segregation
// D = Dependency Inversion
Analogi: SOLID itu kayak aturan main tim futsal. Kalau semua paham peran, permainan jadi kompak.

A.2 Why SOLID Matters in Software Development

Tanpa SOLID, code cepat jadi spaghetti. Sekali ubah satu bagian, bagian lain bisa ikut rusak.

// anti-SOLID: semua kerjaan numpuk di satu class
class AppManager
{
    public void SaveData() {}
    public void SendEmail() {}
    public void PrintReport() {}
    public void ProcessPayment() {}
}
Analogi: kalau satu orang pegang semua kerjaan warung, pas dia sakit langsung chaos.

A.3 History of SOLID Principles

Prinsip ini dipopulerkan Robert C. Martin (Uncle Bob), lalu jadi fondasi di banyak pendekatan clean architecture.

// mindset SOLID: desain dulu tanggung jawabnya,
// baru implementasi detailnya.
Console.WriteLine("Design before coding");
Analogi: sebelum bangun rumah, gambar denah dulu biar gak bongkar pasang tembok.

A.4 Benefits of Applying SOLID

Manfaat paling kerasa: maintainable, testable, scalable, dan onboarding tim baru jadi lebih cepat.

var manfaat = new List<string>
{
    "Code lebih rapi",
    "Bug lebih gampang dilacak",
    "Feature baru lebih aman ditambah"
};
foreach (var m in manfaat) Console.WriteLine(m);
Analogi: rak toko yang rapi bikin stok gampang dicari dan update harga lebih cepat.

A.5 SOLID in Object-Oriented Programming

SOLID paling cocok dipakai di OOP karena fokusnya pada class, interface, dan hubungan antar objek.

interface IService
{
    void Execute();
}

class OrderService : IService
{
    public void Execute() => Console.WriteLine("Process order");
}
Analogi: OOP itu kayak organisasi tim, SOLID itu SOP biar timnya jalan mulus.

BAB B. Single Responsibility Principle (SRP)

B.1 Definition of SRP

SRP bilang satu class cuma boleh punya satu alasan untuk berubah. Satu class, satu tanggung jawab utama.

class InvoicePrinter
{
    public void Print(string data)
    {
        Console.WriteLine(data);
    }
}
Analogi: kasir ya fokus bayar-membayar, bukan sekalian bikin menu dapur.

B.2 Why Single Responsibility Matters

Kalau tanggung jawab campur aduk, perubahan kecil bisa nyeret banyak bug ke area lain.

// lebih aman dipisah
class InvoiceRepository { public void Save() {} }
class InvoiceEmailSender { public void Send() {} }
Analogi: tugas piket dibagi jelas bikin rumah kontrakan tetap bersih tanpa drama.

B.3 Common Violations of SRP

Pelanggaran umum: class service yang isinya validasi, query DB, format PDF, kirim email sekaligus.

class MegaService
{
    public void Validate() {}
    public void SaveToDb() {}
    public void BuildPdf() {}
    public void SendEmail() {}
}
Analogi: satu orang jadi MC, soundman, kameramen, sama panitia konsumsi bareng-bareng.

B.4 Refactoring Code Using SRP

Refactor dengan pecah class berdasarkan role, terus kasih nama class yang jelas.

class OrderValidator { public bool IsValid() => true; }
class OrderRepository { public void Save() {} }
class OrderNotifier { public void NotifyUser() {} }
Analogi: pecah divisi event jadi acara, konsumsi, dokumentasi.

B.5 SRP Implementation Example

Contoh implementasi: flow checkout dipisah per langkah biar gampang ditest.

if (new OrderValidator().IsValid())
{
    new OrderRepository().Save();
    new OrderNotifier().NotifyUser();
}
Analogi: proses jualan punya alur: cek stok, proses bayar, kirim nota.

BAB C. Open-Closed Principle (OCP)

C.1 Definition of OCP

Open-Closed Principle bilang bahwa software sebaiknya mudah ditambah perilaku baru tanpa harus sering mengubah kode inti yang sudah jalan stabil. Intinya bukan berarti kode sama sekali tidak boleh diubah, tapi bagian core yang sudah dipakai banyak tempat harus dijaga supaya perubahan fitur baru tidak memicu efek samping ke fitur lama.

Kalau tiap ada promo baru, tipe notifikasi baru, atau metode pembayaran baru lo selalu membuka class lama lalu menambah if-else baru, itu tanda desainnya belum benar-benar mengikuti OCP. Prinsip ini mendorong kita membuat titik ekstensi yang jelas, biasanya lewat interface atau abstraction.

public interface IDiscountStrategy
{
  decimal CalculateDiscount(decimal totalAmount);
}
Analogi: warung yang sehat itu bisa nambah menu baru lewat resep baru, bukan tiap minggu bongkar dapur utama.

C.2 Understanding Extension vs Modification

Perbedaan paling penting di OCP adalah antara modification dan extension. Modification berarti lo membongkar kode lama untuk menambahkan aturan baru. Extension berarti lo cukup menambahkan komponen baru yang mengikuti kontrak yang sudah ada.

Misalnya checkout awalnya hanya punya diskon regular. Lalu bisnis minta diskon VIP, diskon member baru, dan diskon akhir tahun. Kalau semua aturan itu ditaruh di satu method pakai banyak kondisi, class tersebut akan makin panjang, makin sulit dites, dan makin berisiko rusak saat diubah.

// Buruk: setiap promo baru harus mengubah class yang sama.
public class DiscountCalculator
{
  public decimal Calculate(string customerType, decimal totalAmount)
  {
    if (customerType == "Regular")
      return totalAmount * 0.05m;

    if (customerType == "Vip")
      return totalAmount * 0.15m;

    if (customerType == "NewMember")
      return totalAmount * 0.10m;
    return 0m;
  }
}

Desain yang lebih sehat adalah membuat satu kontrak diskon, lalu tiap aturan diskon menjadi class sendiri. Saat promo baru datang, lo cukup tambahkan class baru tanpa menyentuh proses checkout yang sudah stabil.

public class RegularDiscount : IDiscountStrategy
{
  public decimal CalculateDiscount(decimal totalAmount)
  {
    return totalAmount * 0.05m;
  }
}

public class VipDiscount : IDiscountStrategy
{
  public decimal CalculateDiscount(decimal totalAmount)
  {
    return totalAmount * 0.15m;
  }
}
Analogi: tambah cabang usaha baru harusnya cukup buka tempat baru dengan SOP yang sama, bukan tiap minggu ngerombak sistem kasir pusat.

C.3 Using Abstraction for OCP

Abstraksi adalah kunci OCP karena abstraction membuat kode level atas bergantung pada kontrak, bukan implementasi detail. Class utama seperti checkout, payment, atau notification service cukup tahu bahwa ada sesuatu yang bisa menghitung diskon. Dia tidak perlu tahu apakah diskon itu regular, VIP, seasonal, atau campaign tertentu.

Begitu high-level module hanya mengenal interface, perilaku baru bisa disuntikkan dari luar. Ini bikin code lebih fleksibel, testable, dan aman dikembangkan.

public class CheckoutService
{
  private readonly IDiscountStrategy _discountStrategy;

  public CheckoutService(IDiscountStrategy discountStrategy)
  {
    _discountStrategy = discountStrategy;
  }

  public decimal CalculateFinalPrice(decimal totalAmount)
  {
    var discount = _discountStrategy.CalculateDiscount(totalAmount);
    return totalAmount - discount;
  }
}

Dengan desain ini, CheckoutService tidak perlu diubah saat muncul jenis diskon baru. Selama class baru tetap mengikuti IDiscountStrategy, service ini tetap bisa dipakai tanpa modifikasi.

Analogi: lo pesan lewat admin toko. Lo gak perlu tahu di dapur mereka pakai kompor gas atau listrik, yang penting hasil pesanannya sesuai.

C.4 Strategy Pattern and OCP

Strategy Pattern adalah salah satu pattern paling umum untuk menerapkan OCP. Ide dasarnya sederhana: perilaku yang bisa berubah dipindahkan ke object strategy. Class utama cukup memakai strategy yang aktif saat itu.

Pola ini cocok untuk aturan diskon, pajak, ongkir, format export, validasi, dan banyak variasi business rule lain. Daripada menaruh semua logika di satu method besar, lo pecah menjadi strategi-strategi kecil yang fokus.

public class NewMemberDiscount : IDiscountStrategy
{
  public decimal CalculateDiscount(decimal totalAmount)
  {
    return totalAmount * 0.10m;
  }
}

public class NoDiscount : IDiscountStrategy
{
  public decimal CalculateDiscount(decimal totalAmount)
  {
    return 0m;
  }
}

Kalau nanti bisnis minta HolidayDiscount, lo tinggal tambah class baru. Tidak ada alasan untuk mengedit checkout service selama kontraknya tetap sama. Itu esensi OCP: fitur bertambah lewat ekstensi, bukan tambal-sulam di core logic.

Analogi: diskon member biasa, VIP, dan promo akhir tahun boleh beda aturan, tapi mesin kasir tetap jalan dengan cara yang sama.

C.5 OCP Implementation Example

Di bawah ini contoh implementasi OCP yang lebih utuh. Perhatikan bahwa CheckoutService tidak pernah tahu detail jenis diskon. Ia hanya menjalankan kontrak yang diberikan. Saat aturan baru muncul, kita menambah strategy baru tanpa memodifikasi service utama.

using System;

public interface IDiscountStrategy
{
  decimal CalculateDiscount(decimal totalAmount);
}

public class RegularDiscount : IDiscountStrategy
{
  public decimal CalculateDiscount(decimal totalAmount)
  {
    return totalAmount * 0.05m;
  }
}

public class VipDiscount : IDiscountStrategy
{
  public decimal CalculateDiscount(decimal totalAmount)
  {
    return totalAmount * 0.15m;
  }
}

public class HolidayDiscount : IDiscountStrategy
{
  public decimal CalculateDiscount(decimal totalAmount)
  {
    return totalAmount * 0.20m;
  }
}

public class CheckoutService
{
  private readonly IDiscountStrategy _discountStrategy;

  public CheckoutService(IDiscountStrategy discountStrategy)
  {
    _discountStrategy = discountStrategy;
  }

  public decimal CalculateFinalPrice(decimal totalAmount)
  {
    var discount = _discountStrategy.CalculateDiscount(totalAmount);

    return totalAmount - discount;
  }
}

public class Program
{
  public static void Main()
  {
    decimal totalAmount = 1_000_000m;

    var regularCheckout = new CheckoutService(new RegularDiscount());
    var vipCheckout = new CheckoutService(new VipDiscount());
    var holidayCheckout = new CheckoutService(new HolidayDiscount());

    Console.WriteLine($"Regular : {regularCheckout.CalculateFinalPrice(totalAmount)}");
    Console.WriteLine($"VIP     : {vipCheckout.CalculateFinalPrice(totalAmount)}");
    Console.WriteLine($"Holiday : {holidayCheckout.CalculateFinalPrice(totalAmount)}");
  }
}

Kalau minggu depan ada promo FlashSaleDiscount, lo tinggal tambahkan class baru yang implement IDiscountStrategy. CheckoutService tetap aman, test lama tetap relevan, dan risiko regresi lebih kecil.

Analogi: nambah paket bundling baru di toko harusnya cukup tambah aturan paketnya, bukan bongkar ulang semua rak dan meja kasir.

BAB D. Liskov Substitution Principle (LSP)

D.1 Definition of LSP

Liskov Substitution Principle bilang bahwa object turunan harus bisa dipakai sebagai pengganti object induknya tanpa merusak ekspektasi perilaku dari kode yang memakainya. Jadi, warisan class bukan cuma soal punya method yang sama, tapi juga soal menjaga janji behavior yang konsisten.

Kalau sebuah method menerima object bertipe parent, method itu seharusnya tetap bekerja normal saat diberi object child. Kalau object child membuat hasilnya berubah drastis, melempar exception aneh, atau melanggar aturan bisnis yang sebelumnya aman, berarti desainnya melanggar LSP.

public abstract class Bird
{
  public abstract void Move();
}

public class Sparrow : Bird
{
  public override void Move()
  {
    Console.WriteLine("Sparrow moves by flying.");
  }
}
Analogi: pemain cadangan yang masuk harus tetap bisa menjalankan peran tim sesuai skema pelatih, bukan bikin strategi tim ambruk.

D.2 Inheritance and Behavioral Substitution

Banyak orang salah paham bahwa selama subclass punya method yang sama, berarti aman dipakai sebagai pengganti parent. Padahal yang diuji oleh LSP adalah behavior, bukan sekadar bentuk API. Kontrak parent biasanya membawa ekspektasi implisit: method harus valid dipanggil, hasilnya masuk akal, dan state object tetap konsisten.

Artinya, saat kita mendesain inheritance, kita harus bertanya: "Kalau object ini dipakai di semua tempat yang menerima parent class, apakah hasilnya tetap benar?" Kalau jawabannya tidak, maka relasi parent-child itu mungkin salah secara desain.

public static class BirdDemo
{
  public static void MakeBirdMove(Bird bird)
  {
    bird.Move();
  }

  public static void Run()
  {
    Bird sparrow = new Sparrow();
    MakeBirdMove(sparrow);
  }
}

Contoh di atas aman karena Sparrow memenuhi ekspektasi bahwa semua Bird memang bisa dipanggil method Move() dan hasilnya tetap valid.

Analogi: siapa pun yang pegang shift kasir harus tetap bisa menjalankan alur kasir yang sama. Boleh gaya kerjanya beda, tapi fungsi dasarnya tidak boleh rusak.

D.3 Common LSP Violations

Violation LSP paling umum terjadi saat subclass memaksa parent class menanggung behavior yang tidak cocok. Contoh klasiknya adalah membuat parent Bird punya method Fly(), lalu semua turunan dipaksa mengikutinya. Masalahnya, tidak semua burung bisa terbang. Akhirnya subclass seperti penguin bisa melempar exception atau mengisi method dengan perilaku palsu.

Masalah seperti ini berbahaya karena kode di level atas percaya bahwa semua object parent aman dipakai dengan cara yang sama, padahal kenyataannya tidak. Ini bisa menghasilkan bug yang baru terlihat saat runtime.

public abstract class BirdWithFlight
{
  public abstract void Fly();
}

public class Eagle : BirdWithFlight
{
  public override void Fly()
  {
    Console.WriteLine("Eagle is flying high.");
  }
}

public class Penguin : BirdWithFlight
{
  public override void Fly()
  {
    throw new NotSupportedException("Penguin cannot fly.");
  }
}

Di sini Penguin memang secara teknis mewarisi BirdWithFlight, tapi secara behavior dia tidak bisa menggantikan parent dengan aman. Begitu client code memanggil Fly(), program rusak. Itu pelanggaran LSP.

Analogi: lo rekrut orang buat posisi driver, tapi pas disuruh bawa mobil dia bilang "saya sebenarnya gak bisa nyetir". Secara CV mungkin mirip, tapi secara fungsi gagal total.

D.4 Refactoring Code to Follow LSP

Cara memperbaiki pelanggaran LSP adalah mendesain abstraction yang lebih jujur terhadap kemampuan object. Jangan paksa semua turunan mewarisi perilaku yang sebenarnya tidak relevan. Pisahkan kontrak berdasarkan kemampuan yang benar-benar dimiliki object tersebut.

Dalam kasus burung, kita bisa punya abstraction umum untuk bergerak, lalu interface terpisah untuk kemampuan terbang. Dengan begitu, burung yang bisa terbang implement IFlyable, sedangkan penguin tidak dipaksa pura-pura bisa terbang.

public interface IMovable
{
  void Move();
}

public interface IFlyable
{
  void Fly();
}

public class Sparrow2 : IMovable, IFlyable
{
  public void Move()
  {
    Console.WriteLine("Sparrow hops on the branch.");
  }

  public void Fly()
  {
    Console.WriteLine("Sparrow flies.");
  }
}

public class Penguin2 : IMovable
{
  public void Move()
  {
    Console.WriteLine("Penguin walks and swims.");
  }
}

Setelah dipisah seperti ini, client code yang butuh kemampuan terbang hanya menerima IFlyable, sedangkan logic umum pergerakan cukup memakai IMovable. Kontrak jadi lebih presisi dan substitusi tetap aman.

Analogi: jangan paksa semua anggota tim jadi public speaker. Ada yang jago presentasi, ada yang jago analisis, dan desain role yang bagus menghormati kemampuan itu.

D.5 LSP Implementation Example

Contoh implementasi berikut menunjukkan desain yang lebih aman. Kita tidak lagi memaksakan semua class ke kontrak yang salah. Sebaliknya, kita membuat kontrak yang sesuai dengan perilaku object, sehingga setiap object bisa benar-benar menggantikan abstraction yang dia implement.

using System;
using System.Collections.Generic;

public interface IMovable
{
  void Move();
}

public interface IFlyable
{
  void Fly();
}

public class Sparrow : IMovable, IFlyable
{
  public void Move()
  {
    Console.WriteLine("Sparrow moves by hopping.");
  }

  public void Fly()
  {
    Console.WriteLine("Sparrow flies through the air.");
  }
}

public class Penguin : IMovable
{
  public void Move()
  {
    Console.WriteLine("Penguin moves by walking and swimming.");
  }
}

public static class BirdTrainer
{
  public static void MakeAllBirdsMove(IEnumerable<IMovable> birds)
  {
    foreach (var bird in birds)
    {
      bird.Move();
    }
  }

  public static void StartFlyingSession(IEnumerable<IFlyable> flyingBirds)
  {
    foreach (var bird in flyingBirds)
    {
      bird.Fly();
    }
  }
}

public class Program
{
  public static void Main()
  {
    var movableBirds = new List<IMovable>
    {
      new Sparrow(),
      new Penguin()
    };

    var flyingBirds = new List<IFlyable>
    {
      new Sparrow()
    };

    BirdTrainer.MakeAllBirdsMove(movableBirds);
    BirdTrainer.StartFlyingSession(flyingBirds);
  }
}

Desain ini mengikuti LSP karena semua object dalam koleksi IMovable benar-benar bisa bergerak dengan aman, dan semua object dalam koleksi IFlyable benar-benar bisa terbang. Client code tidak lagi dibohongi oleh inheritance yang terlalu luas.

Analogi: tim campuran tetap jalan lancar kalau role-nya dirancang sesuai kemampuan asli tiap orang, bukan asal semua dimasukin ke jobdesc yang sama.

BAB E. Interface Segregation Principle (ISP)

E.1 Definition of ISP

ISP bilang client gak boleh dipaksa implement method yang dia gak butuh.

interface IPrinter { void Print(); }
interface IScanner { void Scan(); }
Analogi: jangan paksa barista juga wajib ngerti servis AC.

E.2 Problems with Fat Interfaces

Interface kegedean bikin class implement method kosong atau lempar exception.

interface IMachine
{
    void Print();
    void Scan();
    void Fax();
}
Analogi: jobdesc terlalu panjang bikin orang kerja setengah hati di tugas yang bukan bidangnya.

E.3 Designing Smaller Interfaces

Pecah interface jadi kecil-kecil sesuai use case nyata.

class BasicPrinter : IPrinter
{
    public void Print() => Console.WriteLine("Printing...");
}
Analogi: tim konten, tim desain, tim ads dipisah supaya fokus.

E.4 Benefits of Interface Segregation

Dampaknya: coupling turun, testing lebih gampang, implementasi jadi lebih presisi.

class MultiFunctionDevice : IPrinter, IScanner
{
    public void Print() => Console.WriteLine("Print done");
    public void Scan() => Console.WriteLine("Scan done");
}
Analogi: orang kerja sesuai skill bikin output lebih cepat dan kualitas lebih stabil.

E.5 ISP Implementation Example

Di project, start dari interface kecil, gabungkan hanya kalau memang dibutuhin by feature.

IPrinter printer = new BasicPrinter();
printer.Print();
Analogi: bikin paket layanan modular, pelanggan tinggal pilih yang dia butuh.

BAB F. Dependency Inversion Principle (DIP)

F.1 Definition of DIP

DIP bilang modul high-level gak boleh bergantung ke modul low-level langsung. Keduanya harus bergantung ke abstraksi.

interface IMessageSender { void Send(string msg); }
Analogi: manajer kasih arahan ke tim lewat SOP, bukan ngatur kabel mesin langsung.

F.2 High-Level vs Low-Level Modules

High-level berisi aturan bisnis, low-level detail teknis kayak email, file, database.

class EmailSender : IMessageSender
{
    public void Send(string msg) => Console.WriteLine($"Email: {msg}");
}
Analogi: owner fokus strategi jualan, teknisi fokus alat kasir.

F.3 Using Abstraction and Interfaces

Dengan interface, lo bisa ganti implementasi tanpa ubah logic utama.

class NotificationService
{
    private readonly IMessageSender _sender;
    public NotificationService(IMessageSender sender) { _sender = sender; }
    public void Notify(string m) => _sender.Send(m);
}
Analogi: bisa kirim info via WA atau email, tapi format pengumuman tim tetap sama.

F.4 Dependency Injection Concepts

Dependency Injection itu teknik masukin dependency dari luar class, bukan class-nya yang bikin sendiri.

var service = new NotificationService(new EmailSender());
service.Notify("Pesanan siap kirim");
Analogi: chef gak perlu beli bahan sendiri, gudang yang supply sesuai kebutuhan menu.

F.5 DIP Implementation Example

Saat pindah channel notifikasi, cukup ganti implementasi interface.

class SmsSender : IMessageSender
{
    public void Send(string msg) => Console.WriteLine($"SMS: {msg}");
}

var svc = new NotificationService(new SmsSender());
svc.Notify("Jadwal berubah");
Analogi: ganti kurir ekspedisi tanpa ngubah proses checkout toko.

BAB G. Applying SOLID in Real Projects

G.1 SOLID in Layered Architecture

Di arsitektur berlapis, SOLID bantu tiap layer punya job jelas dan minim ketergantungan silang.

// Presentation - Application - Domain - Infrastructure
Console.WriteLine("Pisah layer bikin maintainable");
Analogi: dapur, kasir, gudang dipisah area biar operasional gak tabrakan.

G.2 SOLID with Dependency Injection

DI container bikin wiring dependency lebih rapi dan konsisten di project besar.

// .NET style (konsep)
// services.AddScoped<IMessageSender, EmailSender>();
// services.AddScoped<NotificationService>();
Analogi: jadwal shift otomatis, tinggal daftar orang per peran.

G.3 SOLID in Clean Architecture

Clean Architecture cocok banget sama SOLID karena boundary antar layer dijaga ketat.

// Rule utama: dependency mengarah ke dalam (domain)
Console.WriteLine("Domain harus independen dari framework");
Analogi: resep inti warung harus tetap bisa dipakai walau lokasi tokonya pindah.

G.4 Common Mistakes When Using SOLID

Kesalahan umum: over-engineering, bikin interface kebanyakan padahal skala project masih kecil.

// warning sederhana
bool projectKecil = true;
if (projectKecil) Console.WriteLine("Gunakan SOLID secukupnya");
Analogi: jualan kopi pinggir jalan gak perlu SOP setebal buku pabrik.

G.5 Refactoring Legacy Code with SOLID

Refactor legacy pelan-pelan: mulai dari area paling sering berubah dan paling rawan bug.

// langkah aman
// 1) tambah test
// 2) refactor kecil
// 3) retest
Console.WriteLine("Refactor bertahap");
Analogi: renovasi rumah ditempati harus per kamar, bukan langsung bongkar total.

BAB H. SOLID in Modern Development

H.1 SOLID in Backend Development

Backend API yang pakai SOLID biasanya lebih gampang scaling dan lebih aman saat fitur bertambah.

class OrderController
{
    // delegasi ke service, bukan logic berat di controller
}
Analogi: frontliner nerima order, dapur yang proses masak.

H.2 SOLID in Microservices

Di microservices, SOLID bantu tiap service tetap fokus dan gak saling ketergantungan berlebihan.

// Service A: Payment
// Service B: Shipping
// komunikasi via API/event
Analogi: tiap booth festival jual produk masing-masing tapi tetap terhubung kas pusat.

H.3 SOLID in API Design

Desain API yang SOLID biasanya endpoint jelas, contract stabil, dan mudah di-versioning.

// GET /api/orders/{id}
// POST /api/orders
// PATCH /api/orders/{id}/status
Analogi: menu restoran yang jelas bikin pelanggan gak salah pesan.

H.4 SOLID with Design Patterns

Design pattern kayak Strategy, Factory, Adapter sering jadi kendaraan buat nerapin SOLID.

// Strategy + OCP
IDiscountStrategy strategy = new RegularDiscount();
Console.WriteLine(strategy.Calculate(100000));
Analogi: pattern itu kayak template kerja yang udah teruji di banyak tim.

H.5 SOLID Best Practices

Pakai prinsip seperlunya, prioritaskan kebutuhan bisnis dan keterbacaan code tim.

var rules = new[] { "Simple", "Readable", "Testable", "Evolvable" };
foreach (var r in rules) Console.WriteLine(r);
Analogi: latihan gym efektif itu konsisten dan teknik benar, bukan asal berat bebannya.

BAB I. Summary

I.1 Recap of the SOLID Principles

Ringkasnya: SRP pisahin tanggung jawab, OCP ekstensi tanpa modif core, LSP jaga substitusi, ISP pecah interface, DIP bergantung ke abstraksi.

// checklist cepat
// SRP OCP LSP ISP DIP
Console.WriteLine("SOLID checklist ready");
Analogi: ini lima jurus dasar biar tim coding gak gampang panik.

I.2 When to Apply SOLID

Terapkan saat project mulai tumbuh, tim makin banyak, dan kebutuhan perubahan makin sering.

bool seringBerubah = true;
if (seringBerubah) Console.WriteLine("Saatnya disiplin SOLID");
Analogi: kalau motor sering dipakai jarak jauh, servis berkala jadi wajib.

I.3 When Not to Overuse SOLID

Jangan overkill di script kecil sekali pakai. Tetap pragmatis, jangan bikin abstraksi demi abstraksi.

bool scriptSederhana = true;
if (scriptSederhana) Console.WriteLine("Keep it simple");
Analogi: beli rak industri buat simpan dua gelas doang itu kebanyakan gaya.

BAB J. Glossary

J.1 Core Terms

Istilah inti yang paling sering muncul saat bahas SOLID:

  • Abstraction
  • Dependency Injection
  • Interface
  • Coupling
  • Cohesion
// contoh abstraksi
interface IRepository
{
    void Save();
}
Analogi: ini kayak kosakata wajib kalau mau ngobrol teknis bareng tim.

J.2 OOP Terms

Istilah OOP pendukung SOLID:

  • Refactoring
  • Inheritance
  • Polymorphism
class Animal { public virtual void Speak() {} }
class Cat : Animal { public override void Speak() { Console.WriteLine("Meow"); } }
Analogi: ngerti istilah ini bikin lo gak salah paham pas code review.

J.3 Practical Recap

Target akhir belajar SOLID itu bukan hafalan teori, tapi kebiasaan ngedesain code yang bersih dan tahan lama.

// fokus harian
// 1. tulis code sederhana
// 2. review pakai SOLID
// 3. refactor kalau perlu
Analogi: kayak latihan musik, rutin dikit-dikit tapi konsisten, hasilnya jauh lebih matang.
Kesimpulan: SOLID itu bukan aturan kaku, tapi kompas biar keputusan desain lo tetap waras saat codebase makin gede. Pakai dengan konteks, jangan fanatik.
